# AoT Map System – Label Collision Handling 도입 계획서 (Planning Document)

**Version**: 8.17.0
**Status**: Planning Stage (No patches applied)

---

## 1. 개요 (Overview)

AoT 서비스는 v8.17.0에서 Leaflet 기반 지도 기능을 통합하였으며, 이에 따라 지도 위에 배치되는 장치 라벨, 텍스트 마커, 상태 정보의 가독성 확보가 중요한 이슈로 떠오르고 있다.

현재 AoT 지도에는 장치 수가 증가하거나 특정 줌 레벨에서 밀도가 높아질 경우, 다음과 같은 문제가 발생한다.
-   라벨이 서로 겹쳐 정보가 보이지 않음
-   활성 장치(active)와 비활성 장치(idle)의 구분이 어려움
-   지도 이동/확대 시 시각적 혼란 발생

이를 해결하기 위해 Leaflet 생태계에서 제공하는 충돌 방지(Label Collision) 기술을 도입하고자 하며, 본 문서는 해당 요구에 맞춘 기능 도입 계획을 제시한다.

---

## 2. 요구사항 정의 (Requirements)

### 2.1 기능 요구사항
1.  지도 위 텍스트/라벨이 서로 겹치지 않도록 자동 배치 기능 필요
2.  **Active(중요)** 라벨은 **Idle** 라벨보다 우선 배치되어야 함
3.  충돌이 발생하는 경우 다음 중 하나로 처리 가능해야 함
    -   라벨 숨김
    -   라벨 위치 재배치
    -   라벨 크기/스타일 동적 조정
4.  줌 레벨 변화와 지도 이동에도 안정적으로 동작해야 함
5.  성능 저하가 없어야 하며 모바일 환경에서도 적용 가능해야 함

## 3. 기술 검토 (Available Approaches)

아래는 사용 가능한 라벨 충돌 방지 기술과 AoT에 적용했을 때의 적합성 비교이다.

### 3.1 `L.LabelTextCollision` 플러그인 (Phase 1 후보)

광범위하게 사용되는 Leaflet 충돌 방지 플러그인.

-   **장점**
    -   대중적으로 검증됨.
    -   마커/라벨 충돌 판단을 캔버스 기반으로 처리하여 속도 빠름.
    -   AoT의 Pill Style Marker에도 적용 가능.
    -   설정이 비교적 간단.
-   **단점**
    -   장기간 유지보수 빈약 (실제 PR이 오래됨).
    -   완전한 자동 재배치 기능 미제공 (주로 “숨김” 기능).
    -   DOM 기반 마커보다 Canvas 기반 마커에서 효과가 더 뛰어남.
-   **적합성 평가**
    -   AoT Pill Markers가 DOM 요소이므로 충돌 판정 정확도 중간 수준.
    -   “중요 장치 우선 표시” 요구사항은 커스텀 우선순위 로직 필요.

### 3.2 `Leaflet.MarkerCluster` + Labeling

기존 클러스터링 시스템 활용.

-   **장점**
    -   매우 안정적, 대규모 장치에도 성능 우수.
    -   충돌 문제를 근본적으로 해결함 (군집화).
-   **단점**
    -   라벨 충돌 자체를 해결하는 것이 아니라 클러스터링으로 해결.
    -   **개별 장치 각각의 상태**를 표시해야 하는 AoT 특성과 충돌 가능.
-   **적합성 평가**
    -   AoT는 개별 장치 상태가 중요하므로 적용 제한적.

### 3.3 Custom Canvas-based Layout Engine (Phase 2 후보)

라벨 위치를 직접 계산하여 재배치하는 방식.

-   **방식**: Bounding box 계산, Priority 기반 숨김 or 재배치.
-   **장점**
    -   모든 AoT 요구사항에 100% 커스터마이징 가능.
    -   Active vs Idle 우선순위 쉽게 통합.
-   **단점**
    -   개발 비용 높음.
    -   성능 최적화 필요.
    -   유지보수 부담 존재.
-   **적합성 평가**
    -   AoT 서비스에 가장 최적화된 장기적 방식.

### 3.4 MapLibre + OpenLayers 등 대체 엔진 (Phase 3 후보)

-   **장점**: 지도 객체 레이어 관리가 매우 강력, 충돌 자동 해결 엔진 내장(OpenLayers).
-   **단점**: AoT 전체 지도 시스템 재구축 필요.
-   **적합성 평가**: 중장기적으로 AoT 지도 시스템 확장 시 고려할 가치 있음.

## 4. 권장 도입 방향 (Recommendation)

### ✔ 단기 도입 (Phase 1): `L.LabelTextCollision` + AoT Custom Priority Logic
-   **특징**: 즉시 도입 가능, 리스크 낮음.
-   **내용**:
    -   플러그인 기본 기능으로 겹치는 라벨 자동 숨김 구현.
    -   AoT 장치 상태에 따라 우선순위 부여 (예: Active=10, Idle=1).
-   **목표**: 당장의 문제(가독성 악화)를 최소한의 리스크로 해결.

### ✔ 중기 도입 (Phase 2): AoT 전용 Collision Engine 개발
-   **특징**: 정교한 UX·UI 제공.
-   **내용**:
    -   라벨 재배치 (좌·우·상·하 동적 위치 변경).
    -   성능 고려한 quad-tree 충돌 탐지.
    -   `L.LabelTextCollision` 대체.

### ✔ 장기 도입 (Phase 3): OpenLayers 또는 MapLibre 전환 검토
-   **특징**: GIS 품질 극대화.
-   **내용**: 전문적인 라벨 렌더링, 3D/지형 기능 확장.
